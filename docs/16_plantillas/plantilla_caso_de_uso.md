# Plantilla caso de uso

## Identificador

Asignar un código o nombre breve y estable.

**Ejemplo:**
`UC-001 Registrar paciente`

## Objetivo

Explicar de forma breve qué busca lograr este caso de uso y por qué tiene valor operativo dentro del sistema.

## Alcance

Indicar a qué subdominio o superficie pertenece.

**Ejemplos:**
- consultorio privado;
- farmacia administrativa;
- farmacia pública.

## Actores

Listar los actores principales que participan.

**Ejemplos:**
- médico;
- asistente;
- usuario administrativo de farmacia;
- visitante externo.

## Precondiciones

Describir qué debe ser cierto antes de ejecutar el caso de uso.

## Flujo principal

Describir los pasos normales del recorrido.

Formato sugerido:

1. acción inicial;
2. interacción del usuario o sistema;
3. validación o decisión relevante;
4. resultado del flujo.

## Flujos alternos o excepciones

Registrar variantes importantes del caso.

**Ejemplos:**
- paciente sin cédula;
- horario ocupado;
- producto incompleto;
- operación rechazada por regla de negocio.

## Datos principales involucrados

Resumir qué información entra, se transforma o se persiste durante el flujo.

## Reglas de negocio relacionadas

Listar reglas del dominio que condicionan este caso de uso.

## Resultado esperado

Describir cómo debe quedar el sistema cuando el flujo termina correctamente.

## Errores o rechazos posibles

Indicar situaciones en las que el caso no puede completarse y por qué.

## Impacto sobre otros componentes

Si aplica, indicar qué partes del sistema se ven afectadas.

**Ejemplos:**
- backend consultorio;
- backend farmacia;
- base de datos;
- desktop;
- storefront.

## Observaciones

Espacio opcional para anotar limitaciones, evolución futura o consideraciones particulares.

## Nota de uso

Esta plantilla sirve para documentar flujos funcionales con suficiente claridad como para ser útiles en dominio, backend, UI y pruebas. No debe convertirse en narración excesiva ni quedarse en títulos vacíos.

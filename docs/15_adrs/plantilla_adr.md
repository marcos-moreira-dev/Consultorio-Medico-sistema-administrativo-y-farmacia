# Plantilla ADR

## Título

Breve, claro y centrado en la decisión.

**Ejemplo:**
`ADR-001 - Separar consultorio y farmacia como contextos funcionales distintos`

## Estado

Indicar uno de los estados sugeridos:

- propuesta;
- aceptada;
- reemplazada;
- descartada.

## Fecha

Registrar la fecha en que se redacta o acepta la decisión.

## Contexto

Describir el problema o tensión que obliga a decidir.

Preguntas útiles:

- ¿qué se necesita resolver?
- ¿qué riesgo o confusión existe si no se decide?
- ¿qué parte del sistema se ve afectada?

## Decisión

Explicar de forma directa qué se decidió.

Debe quedar claro:

- qué opción se adopta;
- cuál será la regla a seguir a partir de ahora.

## Alternativas consideradas

Listar, si aporta valor, otras opciones que se evaluaron.

No hace falta hacer una tesis, pero sí conviene dejar claro que la decisión fue comparada con al menos una alternativa cuando el tema lo amerite.

## Justificación

Explicar por qué la decisión elegida resulta más conveniente para el proyecto.

Preguntas útiles:

- ¿qué ventajas ofrece?
- ¿por qué encaja mejor con la V1?
- ¿qué evita o protege?

## Consecuencias

Anotar efectos esperados de la decisión.

Puede incluir:

- beneficios;
- costos;
- complejidad adicional;
- restricciones futuras;
- documentación o componentes que deberán alinearse con esta decisión.

## Impacto sobre el proyecto

Identificar qué partes del sistema deben respetar esta ADR.

**Ejemplos:**
- base de datos;
- backend consultorio;
- backend farmacia;
- seguridad;
- system design;
- UX;
- despliegue.

## Referencias relacionadas

Opcionalmente, anotar documentos o bloques del proyecto con los que esta ADR se relaciona.

## Nota de uso

Esta plantilla debe mantenerse breve y útil. Una ADR buena no intenta contarlo todo: registra una decisión importante con contexto suficiente para poder entenderla más adelante sin depender de memoria informal.

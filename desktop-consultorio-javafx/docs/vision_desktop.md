# Visión desktop

## Propósito

Definir la identidad del cliente de escritorio del consultorio y el papel que cumple dentro del sistema completo.

## Qué tipo de producto es este

`desktop-consultorio-javafx` es una aplicación **de escritorio**, **institucional**, **operativa**, **modular** y **orientada a trabajo diario**.

No es una landing page. No es un experimento visual futurista. No es un frontend web disfrazado. Debe sentirse como software clásico de escritorio: claro, sobrio, confiable y cómodo para sesiones largas.

## Función principal del componente

Este cliente existe para:

- permitir login interno;
- presentar el shell principal del consultorio;
- exponer módulos administrativos y clínico-operativos de forma clara;
- permitir trabajo rápido con tablas, filtros, formularios y modales;
- visualizar dashboards y reportes;
- reflejar el estado de la sesión, la navegación y el contexto activo.

## Relación con el dominio

El cliente debe verse coherente con un consultorio pequeño de clase media, no con una clínica de lujo ni con un hospital masivo.

Debe transmitir:

- cercanía;
- orden;
- confianza;
- sobriedad;
- eficiencia operativa.

## Identidad ficticia asumida

### Marca del consultorio
**Consultorio Santa Emilia**

### Nombre del software
**Santa Emilia Desktop**

### Eslogan implícito
Software de escritorio claro para operar pacientes, agenda, atención, cobros y reportes.

## Decisiones visuales ya fijadas

### Estilo general
- clásico;
- institucional;
- de escritorio;
- ligeramente inspirado en software tradicional tipo Windows bien cuidado;
- sin modernidad vacía ni exceso decorativo.

### Paleta dominante
- escala de grises;
- blanco;
- concho de vino como color secundario institucional;
- rojo solo como acento puntual;
- naranja, si aparece, muy poco y solo como acento funcional menor.

### Branding
- mínimo;
- logo presente;
- quizá una imagen sutil en login;
- nada de propaganda visual agresiva.

## Naturaleza de la experiencia

La experiencia del usuario debe basarse en:

- shell estable;
- navegación lateral persistente;
- módulo activo muy claro;
- tablas como núcleo operativo;
- formularios limpios;
- modales centrados;
- dashboard sobrio;
- menús clásicos en barra superior;
- atajos de teclado razonables.

## Qué no debe parecer

No debe parecer:

- app de marketing;
- SaaS minimalista vacío;
- demo infantil;
- panel recargado de branding;
- producto médico alarmista saturado de rojo.

## Qué sí debe parecer

Debe parecer:

- software de escritorio serio;
- herramienta de operación diaria;
- producto cuidado;
- sistema administrativo con identidad propia;
- interfaz que puede usarse durante horas sin fatigar visualmente.

## Relación con backend

El desktop no decide reglas críticas del negocio. Debe:

- consumir `backend-consultorio`;
- reflejar respuestas y errores con claridad;
- respetar roles y sesión;
- servir como cara operativa del sistema.

## Resultado esperado

La visión desktop debe dejar claro que este cliente JavaFX es la cara interna y operativa del consultorio, con una identidad visual madura, una composición espacial clara y una experiencia de escritorio realista y bien defendida.
